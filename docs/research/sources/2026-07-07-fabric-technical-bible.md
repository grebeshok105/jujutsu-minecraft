# техническая библия fabric 1.21.8 для ai-агента разработки крупного мода

## версия, рамки и источник истины

эта спецификация нацелена именно на экосистему fabric для minecraft 1.21.8 и опирается на три слоя источников: официальную документацию fabric, yarn javadocs для 1.21.8, а также реальные исходники и примеры из fabric api и связанных репозиториев. это важно, потому что после 1.21.6 mojang начал переводить рендер на двухфазную модель extraction/drawing, а в более новых ветках fabric 26.1 произошли уже отдельные несовместимые переименования под official mappings. для 1.21.8 базой надо считать versioned docs на 1.21.8 и yarn 1.21.8, а не переносить на проект выводы из 26.1 без проверки. citeturn12view1turn12view3turn5search4turn30search19

главный инженерный вывод такой: для большого мода на 1.21.8 нельзя строить архитектуру как “набор регистраций в одном классе”. fabric сам по себе модульный, его docs прямо разводят common и client entrypoints, loom умеет физически разделять source sets по окружениям, а data generation, render state, attachments, entities, recipes и модели уже живут в разных жизненных циклах и разных сторонах исполнения. если это не разнести в проекте, ai-агент очень быстро начнет путать серверный код с клиентским, регистраторы с рендером, runtime-синхронизацию с persisted data, и каждое обновление версии будет больнее, чем должно быть. citeturn32view0turn32view1turn32view2turn36view0turn33search2

в reviewed источниках особенно полезны: project structure и loom options для физического разнесения кода по сторонам, networking для модели payload/register/send/validate, basic rendering concepts и rendering in the world для новой двухфазной графики, block entity renderer и entity tutorial для render state в новых рендерах, data generation pages для server/client json-пайплайна, data attachments для синхронизируемого произвольного состояния, yarn javadocs для `RenderLayer`, `DrawContext`, `VertexConsumer`, `ParticleTextureSheet`, а из практических репозиториев - fabric-example-mod, fabric api, sodium, lambdynamiclights и spark. citeturn32view0turn12view4turn12view1turn12view0turn31view0turn31view1turn32view2turn36view0turn25search4turn40view0turn41view0turn41view2turn16search3turn11search8turn16search0turn17search1turn18search0

## проектная архитектура

для крупного мода на fabric 1.21.8 рекомендован такой каркас проекта:

```text
src/
  main/
    java/<modid>/
      common/
        core/
        bootstrap/
        registry/
        networking/
        abilities/
        data/
        entities/
        animation/
        utilities/
        configs/
      ...
    resources/
      assets/<modid>/
      data/<modid>/
      fabric.mod.json
      <modid>.mixins.json

  client/
    java/<modid>/
      client/
        bootstrap/
        rendering/
        shaders/
        particles/
        animation/
        entities/
        networking/
        registry/
        utilities/
      ...
    resources/
      assets/<modid>/
        shaders/
        particles/
        textures/
        models/
```

почему именно так. `common` должен содержать все, что обязано существовать и на dedicated server, и на client: registries, gameplay systems, codecs, payload definitions, persistent data, item/entity/block logic. это прямо совпадает с fabric-моделью `ModInitializer` как common entrypoint и с `src/main/java` как кодом для обеих сторон. `client` должен содержать только клиентские entrypoints и вещи вроде рендера, screen/hud, particle providers, ber/entity renderers, shader/resource bootstrap и client-only packet handlers, потому что fabric docs и loom отдельно подчеркивают разделение client-specific code и compile-time safety через `splitEnvironmentSourceSets()`. citeturn32view0turn32view1turn15search8

`rendering` нужно выделять отдельно даже внутри client, потому что после 1.21.6 рендер перестал быть просто “вызвал draw и забыл”. теперь у тебя есть extraction phase, drawing phase, собственные `RenderPipeline`, отдельные render states, gui/hud preparation, ручные gpu uploads и render passes. смешивать это с gameplay-логикой нельзя - world data читается в extraction, а в drawing у тебя уже должен быть только immutable/thread-safe render state. citeturn12view0turn12view1turn33search3

`abilities` надо держать отдельно от голой `entities`/`items`, если у мода есть сложные активные способности. причина банальная, но критичная: способность почти всегда пересекает сразу несколько слоев - input на клиенте, prediction/ux local-only, authoritative confirm на сервере, cooldown/resource state в common, packets в networking, иногда animation state и particles на клиенте, плюс persistent/synced state через attachments или entity data. fabric networking docs отдельно настаивают на регистрации payload на обеих сторонах и серверной валидации, а attachments позволяют хранить и синхронизировать произвольное состояние сущностей, чанков, уровней и block entities. citeturn12view4turn14view0turn36view0

`networking` должно жить как самостоятельный пакет, а не быть размазанным по `entities` и `items`. payload registration делается централизованно через `PayloadTypeRegistry.clientboundPlay().register` и `serverboundPlay().register`, обработчики регистрируются отдельно для client и server, а docs прямо показывают, что receiving code обязан переживать `level == null` и валидировать target entity/range на сервере. при большом моде это значит: payload types, codecs, routing, resend rules и tracking policy надо держать отдельно от предмета или способности, которая этот пакет инициирует. citeturn12view4turn14view0turn14view3

`data` должно включать codecs, data attachments, data components, saved state, datagen providers и json-driven definitions. fabric docs делают datagen отдельным entrypoint/run config и прямо выносят recipes, advancements, tags, block models, item models и worldgen в отдельный программный пайплайн. item data теперь живет в data components, а произвольное runtime/persistent state можно цеплять attachment api. это абсолютно другой жизненный цикл, чем у runtime gameplay. citeturn32view2turn32view3turn32view4turn32view5turn34search4turn36view0

`animation` надо выделять, если проект крупный. новые entity tutorials уже используют отдельные entity render states и animation state на сущности, а renderer/model читают именно extracted render-state данные. если анимация останется внутри entity class или внутри renderer ad hoc, ai-агент начнет путать gameplay state, synced state и purely visual state. separation здесь нужен не “для красоты”, а чтобы не смешивать authoritative game state со state, который влияет только на позу/визуализацию. citeturn31view1turn30search4

`entities` и `particles` должны быть раздельны. entity - это game object с registry, attributes, goals, synced data, persistence, server logic. particle - клиентский transient visual объект с factory/provider, sprite json и отдельным `ParticleTextureSheet`. docs по entities и particles буквально описывают два разных мира: сущности регистрируются в registries и attribute registry, а частицы - через particle type + client-side provider registration + particle json texture list. citeturn31view1turn12view5turn41view2

`shaders` должен быть отдельным пакетом и resource-поддеревом, потому что shader integration уже завязан на `RenderPipeline`, samplers, uniforms, texture views, framebuffer handles и post-effect passes. это не “ещё один texture helper”. даже `DrawContext` в yarn имеет pipeline-aware texture methods, а vanilla post shaders явно используют разные sampler inputs и compositing logic. citeturn24search0turn40view0turn21search0turn21search1

`registry` нужен отдельно, потому что fabric docs почти на каждой странице повторяют один и тот же паттерн: items в `ModItems`, blocks в `ModBlocks`, entities в `ModEntityTypes`, block entities в `ModBlockEntities`, attributes в `ModAttributes`. docs прямо рекомендуют отдельные helper-классы и статическую инициализацию с явным `initialize()`/registration method, чтобы код не расползался. для ai-агента это золотое правило: никакой хаотической регистрации из случайных классов. один домен - один registrar. citeturn34search0turn34search2turn31view1turn34search1turn34search8turn15search9turn15search11

`configs` стоит держать изолированно как фасад над конкретным конфиг-решением проекта. fabric docs стандартизируют `fabric.mod.json` как метаданные загрузчика, а gameplay/runtime data в современной экосистеме идет уже через datagen, components, attachments, saved data и screen/menu sync. вывод простой: конфиг-слой должен быть тонким адаптером, чтобы не размазывать прямые зависимости на конкретную config-библиотеку по всему коду. это уже инженерная рекомендация на основе разделения жизненных циклов в fabric docs. citeturn15search2turn32view2turn36view0turn34search4

`utilities` нужны, но только в узком виде. fabric docs отдельно советуют использовать events вместо mixins там, где это возможно, а registries, datagen providers и renderer helpers обычно требуют маленьких адаптеров. utility-пакет должен содержать pure helpers, math, codecs, identifier helpers, debug utils. как только туда попадает gameplay state, networking logic или renderer ownership - пакет превратился в свалку. ну классика жанра, да. citeturn15search1turn33search12

## рендеринг fabric 1.21.8

главный факт по 1.21.8 такой: mojang уже перевел chunk/gui/hud rendering на раздельные extraction и drawing этапы, а конечная цель этой перестройки - уметь готовить следующий кадр, пока предыдущий еще рисуется. fabric docs прямо называют это ключевым изменением после 1.21.6, а в 1.21.8 отдельно отмечают, что gui rendering уже adopted this model. из этого следует жесткое архитектурное правило: во время extraction читаем мир и собираем render state, а во время drawing не лезем назад в mutable gameplay state. citeturn12view1turn12view3turn12view0

`RenderPipeline` в yarn - это базовое описание того, чем и как рисовать: шейдеры, формат вершин, режим вершин, bindable uniforms/state, а в world-rendering docs custom pipeline создается через `RenderPipeline.builder(...)` и регистрируется через `RenderPipelines.register(...)`. пример из официальной документации: pipeline, похожий на debug filled box, но без depth stencil state, чтобы боксы были видны сквозь стены. это правильная точка входа для любого нестандартного world renderer, если vanilla pipeline вам не подходит. citeturn26view0turn24search0

`RenderState` в 1.21.8 - не абстрактная “штука где-то внутри ваниллы”, а практически обязательный строительный блок. fabric docs прямо говорят, что render states должны быть immutable, thread-safe и быстрыми в создании, потому что они переживают границу между extraction и drawing. в gui это `GuiRenderState`, в block entity/entity rendering - собственные state classes, в world rendering - ваш extracted state вроде waypoint/render instance list. если в draw-фазе нужен кусок world data, его надо перенести в render state заранее. citeturn33search3turn12view1turn31view0turn31view1turn40view0

`VertexFormat` определяет, какие элементы лежат в вершине и в каком порядке они отправляются в renderer/openGL. official fabric docs перечисляют стандартные форматы `BLOCK`, `NEW_ENTITY`, `PARTICLE`, `POSITION_COLOR`, `POSITION_TEX_COLOR_NORMAL` и другие, а также draw modes `LINES`, `LINE_STRIP`, `DEBUG_LINES`, `TRIANGLES`, `TRIANGLE_STRIP`, `TRIANGLE_FAN`, `QUADS`. критичный нюанс: `VertexConsumer` и `BufferBuilder` обязаны получать элементы строго в порядке формата, иначе интерпретация буфера ломается. citeturn42view1turn41view0

`BufferBuilder` в reviewed docs описан как класс, который форматирует и загружает rendering data в openGL, а `Tesselator`/`Tesselator.getInstance()` - как singleton, из которого buffer builder обычно и стартует в более старом/high-level пути. в новой world-rendering документации custom pipeline пример уже идет глубже: `BufferBuilder` копит вершины, потом строится `MeshData`, затем данные руками копируются в `GpuBuffer` через `CommandEncoder.mapBuffer`, после чего выполняется `RenderPass`. это значит, что для крупного мода надо мыслить не “нарисовать сейчас”, а “собрать буфер, построить draw state, загрузить на gpu, потом отрендерить”. citeturn42view1turn27view0turn27view2turn27view3

`VertexConsumer` в yarn - это интерфейс потребления вершин в заданном формате. он принимает position/color/light/normal/overlay и умеет, например, записывать готовые `BakedQuad`. важная инженерная граница тут такая: `VertexConsumer` - это контракт записи вершины, а `BufferBuilder` - один из concrete implementors этого контракта. если агент пишет низкоуровневый рендер, он должен понимать, когда ему нужен просто `VertexConsumer` от существующего пайплайна, а когда нужно самому владеть `BufferBuilder` и всем upload path. citeturn41view0turn41view1

`RenderLayer` в yarn по-прежнему остается важным понятием, хотя низкоуровнево мир уже едет в сторону `RenderPipeline`. javadocs определяют `RenderLayer` как набор состояний рендера: vertex format, draw mode, shader program, texture, uniform values и gl state вроде depth test, которые layer устанавливает перед рисованием и сбрасывает после. practically это значит: `RenderLayer` - удобный ванильный контракт совместимости и категоризации рендера, а `RenderPipeline` - более “железный” путь для новых систем и custom world passes. не пытайся свалить их в одну ментальную кучу. citeturn25search4

`DrawContext` - центр gui/hud-подготовки в 1.21.8. yarn показывает, что он хранит `GuiRenderState`, stack матриц и scissor stack, а методы вроде `drawGuiTexture`, `drawTexture`, `fill`, `drawItem`, `drawTooltip`, `enableScissor/disableScissor` уже pipeline-aware. fabric docs отдельно объясняют, что в gui/hud you usually do not write raw `BufferBuilder` yourself: `GuiGraphicsExtractor`/`DrawContext` добавляют элемент в gui render state, а upload в `BufferBuilder` происходит позже, в конце preparation. для ai-агента это означает: gui код надо писать декларативно, не тащить world-pipeline idioms в экранный код. citeturn40view0turn28view0turn42view1

в world rendering официальная fabric-страница показывает полный кастомный путь. extraction phase читает данные мира и складывает их в собственный immutable state. drawing phase берет его, при необходимости создает `BufferBuilder` с форматом и draw mode пайплайна, пишет вершины, `buildOrThrow()` превращает это в `MeshData`, при translucent quads возможна сортировка `sortQuads(...)`, дальше вершины копируются в `MappableRingBuffer`, после чего `RenderPass` получает pipeline, default uniforms, dynamic transforms, vertex buffer и index buffer, и вызывает `drawIndexed(...)`. это по сути canonical reference implementation для нестандартного world render pass на 1.21.8. citeturn12view0turn26view1turn26view3turn27view0turn27view2turn27view3turn27view4

`MatrixStack`/`PoseStack` и hud-специфичный `Matrix3x2fStack` нужны не просто для “переместить штуку”. docs отдельно предупреждают, что стек надо push/pop балансировать, иначе получишь broken matrix stack и дальнейшие рендер-артефакты. для hud начиная с 1.21.8 используется `Matrix3x2fStack`, а для 3d-рендера и block/entity renderers по-прежнему важен `PoseStack` с translate/scale/quaternion rotations. block entity renderer docs прямо рекомендуют пушить матрицу в начале `submit` и попать в конце. citeturn42view0turn42view2turn31view0

с практической точки зрения собственная архитектура рендера должна выглядеть так: gameplay code порождает domain state, extraction adapters превращают его в render state, renderer batches агрегируют однотипные примитивы по pipeline/material, upload layer владеет ring buffers/gpu buffers, а draw executors вызываются в правильном world/gui event hook. fabric docs отдельно говорят, что events часто заменяют mixins ради совместимости, а world rendering page показывает hooks уровня `END_EXTRACTION` и `AFTER_TRANSLUCENT_TERRAIN`. это и есть правильный skeleton для большого клиентского рендера. citeturn15search1turn26view1

про tessellation. fabric docs используют термин `Tesselator` как singleton entrypoint к `BufferBuilder`, а далее показывают, как из вершин в `BufferBuilder` получаются фактические triangles/quads/strip’ы в зависимости от `VertexFormat.Mode`. в инженерном смысле tessellation здесь - не отдельная magic subsystem, а процесс, где выбранный draw mode определяет, как последовательность вершин станет примитивами на стороне renderer. на hud docs даже показывают пример `TRIANGLE_STRIP` для ромба. citeturn42view1turn42view0

про dynamic meshes. yarn javadocs для `Model` прямо говорят, что это “dynamic model” со своими render layers и custom rendering, а fabric wiki дополнительно предупреждает: java-side dynamic block/item model обычно рендерится при chunk rebuild; если нужна truly dynamic визуализация, лучше использовать `BlockEntityRenderer`. значит, для крупного мода выбор такой: статический/кэшируемый mesh - через model/baked path, truly dynamic runtime mesh - через ber/custom pipeline. citeturn39search2turn30search10turn31view0

про instancing. в официальных fabric docs и просмотренных yarn/api sources я не нашел high-level публичный api “нажми кнопку и получи generalized instanced rendering” для пользовательских world meshes на 1.21.8. зато есть ясно documented путь через batching, `MappableRingBuffer`, `GpuBuffer`, `RenderPass` и крупные aggregated vertex uploads. поэтому безопасная рекомендация такая: думать не про классический gl-instancing как контракт fabric api, а про instancing-like batching - собирать много логических инстансов в меньшее число больших буферов одного pipeline/material. сам факт отсутствия удобного high-level instancing helper в reviewed docs - это ограничение, а не повод фантазировать. citeturn27view2turn27view3turn24search0

## glsl, постэффекты и кастомные шейдерные паттерны

на базовом уровне glsl в reviewed спецификациях устроен так: vertex shader и fragment shader имеют отдельные пространства имен для функций и глобальных переменных, а общими точками конфигурации служат `uniform`-значения и sampler’ы. в current spec sampler’ы объявляются через `uniform sampler...`, а framebuffer objects позволяют рендерить не только в default framebuffer, но и в пользовательские off-screen targets. это фундамент для всех эффекто-подобных вещей в моде - от простого gui texture pass до outline, blur и прозрачных composite passes. citeturn19search0turn19search4turn19search1turn19search5

реальный 1.21.8 vanilla-пример vertex-stage логики - `rendertype_lines.vsh`. этот shader импортирует fog/globals/dynamictransforms/projection, принимает `Position`, `Color`, `Normal`, считает screen-space направление линии через `ScreenSize`, использует `LineWidth`, а затем смещает `gl_Position` так, чтобы линия имела толщину на экране. это отличный референс по двум вещам сразу: как ваниль использует uniforms/inputs для render pipeline и как переносит вычисление в vertex stage, а не в java-код. citeturn20search2

реальный 1.21.8 post-process пример transparency - `transparency.fsh`. он читает шесть разных sampler’ов и depth-sampler’ов (`Main`, `Translucent`, `ItemEntity`, `Particles`, `Weather`, `Clouds`), собирает активные слои, сортирует их по depth и затем блендит слои функцией `blend(dst, src)`. это очень важный ориентир: в современной ванили прозрачность уже не “нарисуй альфу и надейся”. она может быть explicit composite pass с несколькими targets и depth-aware merge. если вашей способности, броне, aura-эффекту или частицам нужен сложный translucent look, мыслить надо именно в этих терминах. citeturn21search0

реальный 1.21.8 пример пост-краевого эффекта - `entity_sobel.fsh`. шейдер берет `InSampler`, читает соседние тексели по `oneTexel`, считает разницу альфы по сторонам, clamp’ит сумму и формирует итоговый цвет/альфу. это уже прямой референс для outline/highlight/edge-detect и в целом показывает, как в ванили устроен экранный fragment post effect, работающий поверх готового input texture. citeturn21search1

из этого следует практическая схема для кастомной shader-интеграции в моде на 1.21.8. vertex shaders используйте там, где можно дешево перенести geometric/pre-transform work на gpu. fragment shaders - там, где нужна per-pixel логика: маски, край, fog-like falloff, composite, цветовые функции. uniforms - для времени, силы эффекта, матриц, цветовых модификаторов, порогов. samplers - для base color, noise, depth, mask, history/previous pass textures. framebuffer/post targets - когда эффект требует двух и более проходов или работы по уже отрендеренной сцене. эта схема полностью согласуется и с glsl/fbo docs, и с yarn упоминаниями `PostEffectPass`, `PostEffectProcessor`, `Framebuffer`, `GpuTextureView`, и с vanilla shader assets. citeturn19search0turn19search1turn19search5turn24search0turn25search0turn25search2

что с конкретными requested-паттернами. прозрачность и masking подтверждаются источниками напрямую: blending/transparency - через gl blending/fbo model и ванильный transparency composite pass; masking - как fragment-stage контроль видимости через sampled/derived alpha и composite logic. scrolling uv, animated uv, distortion, heat haze, dissolve, fresnel, bloom simulation, signed distance fields и procedural noise - это уже допустимые shader-паттерны на том же фундаменте `uniform + sampler + framebuffer + fragment logic`, но в просмотренных fabric/mojang материалах я не нашел официальных fabric tutorials specifically для 1.21.8, которые бы давали готовые examples именно этих эффектов. безопасно утверждать можно следующее: они реализуются в тех же технических рамках, но конкретный effect-authoring контракт вам придется проектировать самим поверх documented shader/fbo/pipeline primitives. citeturn19search0turn19search1turn19search5turn21search0turn21search1

для “bloom simulation” надежный вывод такой: если нужен glow вокруг объекта, делайте это как минимум двухпроходно - сначала выделение bright/edge/target data в off-screen framebuffer, потом blur/composite назад в целевой framebuffer. сам факт необходимости off-screen render-to-texture подтверждается fbo docs, а vanilla post stack уже использует post samplers и compositing. но готового official fabric recipe “сделай bloom вот так” в reviewed 1.21.8 docs нет. citeturn19search1turn19search5turn21search1turn24search0

для signed distance fields ситуация такая же: reviewed источники дают достаточную glsl-базу, но не дают fabric-specific 1.21.8 tutorial с готовой sdf-пайплайновой реализацией. если мод требует масштабируемые руны, круги способностей или crisp text-like decals, sdf - жизнеспособный подход, но эта часть должна считаться отдельной r&d зоной, а не “из коробки fabric feature”. citeturn19search0turn19search4

## сеть, синхронизация и многопользовательская устойчивость

официальный networking pipeline для fabric 1.21.8 начинается с регистрации payload types на обеих физических сторонах. docs показывают `PayloadTypeRegistry.clientboundPlay().register(...)` и `serverboundPlay().register(...)`, а затем `ServerPlayNetworking.send(...)`/`ClientPlayNetworking.send(...)` для отправки и `ClientPlayNetworking.registerGlobalReceiver(...)`/`ServerPlayNetworking.registerGlobalReceiver(...)` для приема. это должно быть общим инфраструктурным слоем, а не кодом внутри случайного предмета или entity callback. citeturn12view4turn14view0

самое важное правило синхронизации - server authority. official docs буквально показывают, что packet-triggering gameplay action сначала отбрасывает client-side branch через `if (level.isClientSide()) return ...`, а серверный receiver затем валидирует данные пакета: существует ли target entity, находится ли он достаточно близко, подходит ли тип. крупный мод с активными способностями, статусами и кастомными ритуалами должен следовать тому же принципу: клиент может запрашивать и локально предсказывать косметику, но изменение authoritative game state подтверждает сервер. citeturn12view4turn14view0turn14view3

для ability synchronization я рекомендую трехслойную модель. слой первый - input/request packet от клиента на сервер. слой второй - authoritative server state, хранимый либо в entity fields/synched data, либо в data attachments, если состояние должно жить на entity, level, chunk или block entity и, возможно, переживать рестарт мира. слой третий - clientbound cosmetic/feedback packets только для tracking players или для target-only sync policy. эта рекомендация не выдумана с потолка: fabric docs отдельно раскрывают tracking как ключевой концепт экономной сетевой доставки, а data attachments умеют сразу описывать default value, persistence и predicates синхронизации. citeturn14view3turn36view0

для animation synchronization не надо слать “кадр анимации” каждый тик. новые entity docs уже разделяют gameplay entity data, synced entity data и render state. правильнее синхронизировать минимальные семантические факты: состояние способности, старт/стоп анимации, phase token, seed, server tick start, direction/variant. клиент потом сам вычисляет pose/curve локально, а renderer/model читает extracted render state. это радикально уменьшает сетевой шум и делает анимации устойчивее к latency. citeturn31view1turn30search4turn36view0

client prediction в official fabric docs специально не формализована как отдельный модуль, поэтому безопасная инженерная позиция такая: предсказывайте только то, что не ломает правила игры при rollback’е. хорошие кандидаты - локальный звук, слабые preparatory particles, ui cast bar, hand pose. плохие кандидаты - урон, телепорт, реальный cooldown commit, расход ресурса, spawn authoritative entity. docs уже показывают, что обработчик клиента обязан переживать ситуацию `level == null`, а сервер обязан валидировать payload; это косвенно подтверждает, что нельзя делать correctness на клиенте. citeturn14view0turn14view3

по latency handling у мода должен быть не один “send packet”, а набор политик. для одноразовых действий нужен request-id/ability-instance-id, чтобы клиент не применял устаревший confirm к уже отмененному касту. для длительных эффектов нужен phase/state replication, а не поток частичных микродельт каждый кадр. для сущностей и block entities нужен graceful handling ситуации “объект еще не загружен/не отслеживается”. это не абстракция: fabric docs сами показывают проверки на `level == null` и existence of target entity, а fabric api issue tracker отдельно содержит реальные баги вида `AttachmentSyncException disconnects client when target entity is not yet loaded`. citeturn14view0turn14view3turn8search3

про packet ordering. в просмотренных fabric docs я не нашел отдельной спецификации порядка доставки play payloads на уровне fabric api. поэтому безопасный permanent rule такой: не проектировать протокол так, будто два связанных пакета обязаны попасть в “правильный игровой тик” с точки зрения мира/рендера. вместо этого каждый пакет должен быть самодостаточен: иметь identity/phase/version, а обработчик должен уметь отложить применение или отбросить payload, если нужная сущность, мир или attachment state еще не готовы. это ограничение надо принять как спецификацию проекта, а не надеяться на случайную удачу. citeturn14view0turn8search3

многопользовательские edge cases, которые надо считать обязательными: клиентский receiver вызвался, пока уровень еще `null`; target entity уже деспаунен или не загружен; игрок больше не tracking нужный chunk/entity; игрок вышел из радиуса; сервер успел изменить состояние между client prediction и confirm; attachment value еще не успела прийти на клиент; экран/uidraw opened earlier than data arrived. fabric docs и issues достаточно явно показывают хотя бы часть этих проблем, так что архитектура должна быть defensive by default. citeturn14view0turn14view3turn8search3

## частицы и визуальные transient-системы

официальный путь создания кастомной частицы в fabric 1.21.8 состоит из трех этапов: зарегистрировать `ParticleType` в common initializer, зарегистрировать provider/factory на клиенте, и добавить particle json, который указывает набор используемых текстур. docs показывают `FabricParticleTypes.simple()`, `Registry.register(... PARTICLE_TYPE ...)`, а затем на клиенте `ParticleProviderRegistry.getInstance().register(..., EndRodParticle.Provider::new)` и json `"textures": [...]`. это значит, что runtime-класс частицы, client factory и assets описываются раздельно и должны лежать в разных слоях проекта. citeturn12view5

lifecycle частицы в yarn вполне прозрачен: у `Particle` есть поля `age`, `maxAge`, `dead`, `velocityX/Y/Z`, `alpha`, `gravityStrength`, world position и т.д., а метод `tick()` вызывается каждый игровой тик и отвечает за core particle logic, включая движение и коллизию. для безопасной архитектуры это означает: gameplay-система не должна хранить долгоживущую ссылку на конкретный particle instance как на источник истины. частица - это клиентский визуальный runtime object со своим жизненным циклом, а не игровое состояние. citeturn13search0turn13search2

sprite providers и animated particles завязаны на `SpriteBillboardParticle`. yarn javadocs показывают методы `setSprite`, `setSpriteForAge`, а fabric docs требуют particle json со списком textures. practically это значит: “анимированная” частица обычно не посылает по сети каждый frame index, а сам provider/particle выбирает кадр по age/maxAge локально. в large-scale проекте это идеальный компромисс между красивым визуалом и дешевой синхронизацией. citeturn13search0turn13search2turn12view5

key batching rule для particle systems дает `ParticleTextureSheet`. yarn определяет его как record, который задает rendering setup и draw logic на основе требований к depth/textures/transparency, и отдельно говорит: каждый `Particle` возвращает sheet в `getType()`, а при рендере каждый sheet будет drawn once. отсюда прямой performance вывод: если вы плодите много нестандартных visual типов, но можете уложить их в меньшее количество sheet/material categories, система будет батчиться заметно лучше, чем если каждая “похожая” частица получит свой отдельный путь. citeturn41view2turn41view3

прозрачность частиц не надо рассматривать изолированно от остального рендера. vanilla 1.21.8 transparency pass отдельно имеет sampler для `ParticlesSampler` и `ParticlesDepthSampler`, а `ParticleTextureSheet` различает хотя бы `PARTICLE_SHEET_OPAQUE`, `PARTICLE_SHEET_TRANSLUCENT`, `TERRAIN_SHEET`, `CUSTOM` и `NO_RENDER`. инженерный смысл простой: translucent particles - это не просто “alpha < 1”. это отдельная рендер-категория, которая участвует в большом composite pass сцены. злоупотребление ей быстро бьет и по gpu, и по корректности сортировки. citeturn21search0turn41view2turn41view3

custom particle rendering в reviewed материалах документирован частично. yarn явно показывает наличие `ParticleTextureSheet.CUSTOM`, но fabric docs по 1.21.8 дают именно стандартный путь simple particle + provider + textures, без отдельного официального tutorial по полностью кастомному particle render pass. поэтому permanent rule такой: по умолчанию проект должен использовать стандартные opaque/translucent/terrain sheet’ы и sprite-based particles; переход к truly custom particle rendering - это отдельная техническая ветка с повышенным риском совместимости и performance cost. citeturn12view5turn41view2

лучший сетевой дизайн для частиц - не синхронизировать сами частицы, а синхронизировать событие или состояние, из которого клиент локально порождает частицы. fabric networking docs уже демонстрируют модель “сервер рассылает payload игрокам уровня / tracking players, а клиент создает lightning/entity effect локально”. для particle-heavy abilities это почти всегда правильнее: сеть переносит семантику, клиент строит transient visuals сам. citeturn12view4turn14view3

## производительность, масштабирование и анти-паттерны

best practice номер один - сначала профилировать, потом оптимизировать. `spark` прямо позиционируется как performance profiler для minecraft clients/servers/proxies и отдельно рекламирует cpu profiler для лагов, низкого tps и высокой загрузки cpu. visualvm дает sampling/instrumentation для cpu и memory, а jfr встроен в jvm и, по oracle docs, имеет очень низкий overhead, обычно менее 1% на default settings. для большого fabric-мода этого набора достаточно, чтобы разделить симптомы на java cpu, memory allocation, lock/contention и чисто рендерные bottlenecks. citeturn18search0turn18search1turn18search7turn18search13turn18search2turn18search5turn18search20

cpu bottlenecks в fabric-моде обычно растут из трех мест: дорогие тики сущностей/абилок, лишние allocations, и дубль работы между extraction/render/gameplay. docs по рендеру уже подталкивают к правильной форме - extraction state должен быть fast to create, а drawing phase не должна ходить обратно в mutable world data. networking docs подталкивают к tracking-based delivery вместо broadcast всем подряд. значит, правильная оптимизация на cpu - это меньше random world reads в draw-фазе, меньше per-tick packet spam, меньше бесполезных объектов, больше семантической батчизации. citeturn12view0turn33search3turn14view3

gpu bottlenecks обычно приходят из translucent overload, лишних render passes, мелких upload’ов и раздробленных pipeline/material categories. official world-rendering example прямо показывает reuse через `MappableRingBuffer` и общую фазу upload/draw, а vanilla transparency shader показывает, насколько дорогой может стать composite множества слоев. practically это значит: группируйте отрисовку по pipeline/sheet/material, переиспользуйте буферы, избегайте десятков micro-passes ради “красивого эффекта”, особенно если он еще и alpha-heavy. citeturn26view3turn27view2turn27view3turn21search0

allocation reduction и object pooling в этой экосистеме надо понимать прагматично. fabric docs сами показывают долгоживущий allocator и кольцевой vertex buffer в custom render pipeline example, а render state требуют делать быстрым и маленьким. значит, выделять буферы надо редко, а data records/state snapshots делать компактными и short-lived. не надо превращать пуллинг в культ, но и создавать каждый кадр огромные списки, матрицы и transient wrappers там, где достаточно ring buffer / reused collection, тоже не надо. citeturn26view1turn26view3turn27view1turn33search3

entity optimization - это separation of concerns. official entity docs уже советуют держать registries/types отдельно, а render state/animation - отдельно от gameplay state. если сущность содержит pathfinding, ability state, synced data, animation state и renderer helpers в одном классе, вы почти гарантированно получите и сетевой шум, и трудный профайлинг. для серверной производительности это особенно критично: gameplay entity должна знать только то, что нужно для симуляции, а клиентские позы/шейдерные flags должны вычисляться либо через synced semantic flags, либо в render state extraction. citeturn31view1turn34search3

tick optimization для способностей лучше всего достигается не “если еще один giant if в tick()”, а переходом к event- и state-driven модели. fabric docs прямо говорят, что events часто заменяют mixins ради совместимости и performance, а attachments/components/codecs делают данные структурированными. поэтому sustained abilities лучше сводить к компактному state machine + event hooks + time markers, чем к большому количеству независимых every-tick scanners по всем игрокам/чанкам/сущностям. citeturn15search1turn33search12turn36view0turn34search4

network optimization в fabric - почти дословно из docs: используйте tracking, а не global broadcast; регистрируйте payload один раз и переиспользуйте типы; валидируйте serverbound packets; не отправляйте то, что клиент может вывести локально; передавайте semantic delta, а не полный визуальный слепок. это особенно важно для abilities, animation sync и particles. citeturn14view3turn12view4turn14view0

particle optimization: меньше logical types, меньше unique sheets, меньше translucent overdraw, больше local derivation от одного сетевого события, больше sprite-animation over time вместо частой пересборки объекта или постоянных сетевых обновлений. это следует и из particle lifecycle в yarn, и из rule “each sheet will be drawn once”, и из ванильного transparency composite. citeturn13search0turn13search2turn41view2turn21search0

shader optimization: если эффект можно сделать в одном existing composite pass или в одном material/pipeline без extra framebuffer, делайте так. если нужен off-screen target - измеряйте его отдельно spark/jfr/visualvm плюс in-game frame time tools, а не “на глаз вроде норм”. ванильные post shaders уже показывают, что каждый дополнительный sampler/pass - это новая стоимость. citeturn21search0turn21search1turn19search1turn18search0turn18search2

анти-паттерны, которые в fabric 1.21.8 особенно токсичны: raw legacy gl-вызовы вместо minecraft rendering system; client classes в common source set; рендер, который читает мир уже в drawing phase; сетевые протоколы без server validation; payload spam для визуалов, которые можно вывести локально; giant `ModInit` с регистрацией всего на свете; mixin-first подход там, где fabric уже дает event hook; прямое размазывание registry logic по gameplay классам. docs прямо предупреждают против raw opengl, подчеркивают common/client split, и продвигают events ради совместимости. citeturn33search9turn32view0turn32view1turn15search1turn14view0

из high-quality репозиториев полезные ориентиры такие. `fabric-example-mod` - минимальный canonical baseline, годный как sanity check структуры проекта. `fabric-api` - пример модульного api-подхода и источник реальных hooks. `sodium` - эталон того, насколько сильно может отличаться production-grade rendering architecture от учебных примеров, когда цель - fps, stutter и correctness. `lambdynamiclights` - пример клиентской визуальной системы, у которой performance considerations являются частью design. `spark` - стандартный профайлер, который надо включать в эксплуатационный процесс, а не только когда уже все горит. отдельно важно: с sodium 0.6.0+ renderer api support встроен, indium больше не нужен, значит, compatibility strategy крупного рендер-мода должна учитывать современный sodium как основной внешний renderer target. citeturn16search3turn33search2turn16search0turn17search1turn18search0turn16search1

## постоянные правила разработки и ограничения

если этот документ становится “постоянной технической спецификацией” для ai-агента, я бы зафиксировал такие неизменяемые правила.

первое. общий код живет только в `common`, клиентский рендер и ui - только в `client`, а build обязан использовать `splitEnvironmentSourceSets()` для compile-time safety. это не вкусовщина, а прямая защита от ошибочного обращения к client-only api с сервера. citeturn32view0turn32view1

второе. все registries - через domain-specific registrar classes. `ModItems`, `ModBlocks`, `ModEntityTypes`, `ModBlockEntities`, `ModAttributes` и аналогичные для ваших доменов. gameplay code не регистрирует объекты сам. citeturn34search0turn34search2turn34search1turn31view1turn34search8

третье. все сетевые пакеты описываются как typed payload + codec, регистрируются централизованно, а serverbound handlers всегда валидируют вход. “клиент попросил - значит делаем” запрещено. citeturn12view4turn14view0
