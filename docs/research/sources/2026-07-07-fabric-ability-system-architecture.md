# **Архитектура и реализация продвинутой боевой системы Minecraft Fabric (1.21.8): Проект "jujutsumod"**

> **Status: HISTORICAL REFERENCE.** This dated research/design/review record is not the current source of truth. For current behavior use `README.md`, `AGENTS.md`, `SESSION.md`, and `Jujutsu Kaizen/jujutsumod-codebase-codex/00-MOC.md`; current code and tests win on conflict.

Настоящий отчет представляет собой исчерпывающий технический документ (Implementation Bible), описывающий архитектуру, паттерны проектирования и конкретные шаги для реализации продвинутой боевой системы (gameplay, combat, ability system) для мода jujutsumod на базе Minecraft Fabric 1.21.8 и Java 21\. Документ фокусируется на реализации первого функционального среза (vertical slice) персонажа Нобары Кугисаки. Проектирование выполнено с упором на строгую серверную авторитарность (server-authoritative), клиентскую визуализацию (client-only visuals), типизированную сетевую сериализацию (typed payloads) и использование исключительно публичных Fabric API, избегая преждевременного создания монолитных фреймворков и нестабильных хаков на базе Mixin.

## **1\. Регистрация компонентов и управление ресурсами**

В версии Minecraft 1.21.8 и соответствующих сборках Fabric API парадигма регистрации и управления данными претерпела фундаментальные изменения, направленные на дата-ориентированность, безопасность типов и устранение устаревших структур, таких как NBT на предметах.

### **Регистрация предметов и Data Components**

Начиная с версии 1.21.2, процесс регистрации предметов требует обязательного связывания ключа реестра (RegistryKey) с настройками предмета (Item.Settings) до инициализации самого экземпляра предмета1. Это необходимо для корректного сохранения моделей и ключей перевода в компонентах предмета. Классический подход с инициализацией new Identifier() заменен на фабричный метод Identifier.of() или Identifier.fromNamespaceAndPath()1.  
Традиционный формат NBT для хранения данных на предметах полностью заменен на Data Components (структурированные компоненты данных). Компоненты позволяют хранить персистентные данные (например, уровень заряда молотка или количество накопленных гвоздей) с жесткой типизацией и валидацией на этапе компиляции3. Регистрация собственного компонента осуществляется через DataComponentType, который принимает сериализатор (Codec) для обеспечения персистентности и сетевой синхронизации.

| Идентификатор компонента | Тип данных Java | Codec сериализации | Роль в экосистеме Нобары |
| :---- | :---- | :---- | :---- |
| jujutsumod:resonance\_charge | Integer | Codec.INT | Хранит текущий уровень заряда проклятой энергии в предмете. |
| jujutsumod:nail\_count | Integer | Codec.INT | Количество доступных гвоздей, привязанных к конкретному предмету. Инициализируется по умолчанию. |
| jujutsumod:bound\_targets | List\<UUID\> | UUID\_LIST\_CODEC | Список UUID целей, пораженных гвоздями из этого стека, для массового подрыва. |

Использование метода getOrDefault при чтении компонента обеспечивает отказоустойчивость, возвращая безопасное резервное значение при отсутствии компонента на стеке, что предотвращает ошибки во время выполнения (Runtime Exceptions)3.

### **Регистрация команд, звуков и частиц**

Для обеспечения полного цикла обратной связи необходимо зарегистрировать вспомогательные ресурсы. Команды отладки интегрируются через CommandRegistrationCallback с использованием библиотеки Brigadier. Все команды, управляющие боевыми состояниями, выполняются строго на стороне сервера. Звуки и частицы регистрируются в Registries.SOUND\_EVENT и Registries.PARTICLE\_TYPE. В версии 1.21.8 частицы требуют определения StreamCodec для сетевой передачи через RegistryFriendlyByteBuf4. В jujutsumod клиентская визуализация Резонанса использует локальные вызовы партиклов без создания реальных сущностей на сервере.

### **Статусные эффекты, атрибуты и персистентность (Attachment API)**

Для хранения постоянных и временных состояний сущностей (например, меток резонанса на врагах или максимального запаса проклятой энергии игрока) используется Fabric Data Attachment API (AttachmentRegistry)5. Это исключает необходимость внедрения полей в класс Entity через Mixin.  
Аттачменты обладают встроенной функцией сетевой синхронизации. В 1.21.8 для типа аттачмента можно определить политику синхронизации через метод syncWith(), указав StreamCodec и предикат видимости, например AttachmentSyncPredicate.all() или таргетированный предикат, отправляющий данные только самому игроку5. Для боевой системы Нобары маркеры Резонанса регистрируются с флагом copyOnDeath(false) (маркеры исчезают при смерти сущности), в то время как персистентные данные игрока (разблокированные способности) регистрируются с persistent(Codec) и copyOnDeath(true)8.

## **2\. Обработка пользовательского ввода (Player Input)**

Реализация ввода требует точного баланса между клиентской отзывчивостью (Client Prediction) и защитой от манипуляций (Server Authority).

### **Клиентская архитектура Keybinds**

Регистрация горячих клавиш производится с использованием KeyBindingHelper.registerKeyBinding исключительно в клиентской точке входа9. Обработка нажатий осуществляется в событии ClientTickEvents.END\_CLIENT\_TICK. Для способностей с длительным временем применения (кастом) необходимо отслеживать непрерывное состояние isPressed() и точечный момент wasPressedThisFrame().  
Как только клиент фиксирует нажатие, он немедленно активирует визуальную обратную связь (Visual Feedback) — например, поднимает модель молотка или генерирует частицы концентрации проклятой энергии вокруг игрока. Эта иллюзия мгновенного отклика скрывает сетевую задержку. Однако фактический расчет урона, списание проклятой энергии и инициация кулдауна применяются только после подтверждения сервером.

### **Отправка запросов и валидация на сервере**

При нажатии клавиши клиент формирует пакет (Ability Request) и отправляет его на сервер. На стороне сервера происходит строгая валидация. Сервер извлекает профиль игрока и проверяет:

1. Наличие необходимого предмета в руках (если способность привязана к предмету).  
2. Текущее значение аттачмента проклятой энергии (хватает ли ресурсов).  
3. Состояние серверного кулдауна.  
4. Отсутствие блокирующих статусных эффектов (например, оглушения).

### **Защита от спама и некорректных пакетов**

Одной из главных угроз в мультиплеерной среде является пакетный спам, когда модифицированный клиент отправляет тысячи запросов в секунду. Архитектура jujutsumod реализует двухуровневую защиту:

* **Клиентский троттлинг:** Клиент проверяет локальный кулдаун перед отправкой пакета.  
* **Серверный Rate Limiting:** Обработчик пакетов на сервере хранит метку времени (timestamp) последнего полученного пакета для каждого UUID. Если дельта времени между пакетами меньше физически возможной (например, 50 мс), пакеты молча отбрасываются (Drop), а при систематическом нарушении генерируется предупреждение в лог.

### **Синхронизация кулдаунов (Cooldown Sync)**

Сервер выступает единственным источником истины для времени отката способностей. После успешной валидации и применения способности сервер обновляет локальный таймер и отправляет AbilityStateSyncPayload с оставшимся временем в тиках. Клиентская логика использует это значение исключительно для рендеринга GUI (затемнения иконок). При любой рассинхронизации клиентский счетчик принудительно корректируется серверным значением.

## **3\. Сетевое взаимодействие (Networking)**

Архитектура сетевого слоя базируется на модели "Нулевого доверия" (Zero-Trust) к клиенту. Вся бизнес-логика игры рассчитывается на логическом сервере.

### **Типизированные полезные нагрузки (Custom Payload Patterns)**

Начиная с версии 1.21, Fabric полностью перешел на Mojang API для работы с сетью: CustomPacketPayload, CustomPacketPayload.Type и StreamCodec поверх RegistryFriendlyByteBuf9. Это заменяет устаревший PacketByteBuf.  
Каждый сетевой пакет реализуется как Java Record, имплементирующий CustomPacketPayload. Внутри Record определяется статический Type (уникальный идентификатор Identifier) и StreamCodec (обычно собираемый через StreamCodec.composite для композиции нескольких полей)10. Регистрация пакетов происходит в фазе общей инициализации через PayloadTypeRegistry.clientboundPlay().register() и serverboundPlay().register()10.

| Направление | Идентификатор (Type) | Содержимое (Record Fields) | Архитектурное назначение и безопасность |
| :---- | :---- | :---- | :---- |
| C2S | AbilityRequestPayload | int abilityId, boolean isKeyDown | Клиент сообщает о намерении использовать способность. Никогда не передает координаты цели или нанесенный урон. |
| C2S | TargetEntityPayload | int entityId | Используется для способностей, требующих lock-on. Сервер самостоятельно проверяет LoS и дистанцию до этой сущности. |
| S2C | AbilityStateSyncPayload | int abilityId, int cooldownTicks | Синхронизация кулдауна с сервером для отображения UI. Предотвращает рассинхронизацию интерфейса. |
| S2C | HairpinFxPayload | int fxType, List\<Integer\> entityIds | Сервер приказывает клиентам отрисовать визуальные эффекты (Hairpin) на указанных сущностях. Экономит трафик, заменяя создание реальных Entity партиклов. |

### **Размер пакетов, безопасность и данные, не заслуживающие доверия**

Размер пакетов минимизируется за счет передачи примитивов (int для ID способностей и entityId вместо полных UUID или сложных объектов). RegistryFriendlyByteBuf позволяет эффективно сжимать и передавать данные реестров.  
Критическое правило безопасности: **Клиенту нельзя доверять**. Клиент никогда не должен передавать пакеты вида "Я нанес 50 урона сущности X" или "Мой луч попал в координаты Y". Вместо этого клиент отправляет только сам факт нажатия кнопки. Сервер самостоятельно вычисляет вектор взгляда игрока (player.getCameraPosVec()), проводит Raycast, определяет пересечение с хитбоксами и начисляет урон.

## **4\. Архитектура способностей (Ability Framework)**

Для первого среза геймплея Нобары избегается внедрение переусложненных монолитных фреймворков. Применяется минималистичная, но масштабируемая архитектура, основанная на компонентном подходе.

### **Базовые классы и интерфейсы**

Минимальный набор для старта включает следующие структуры:

* IAbility: Интерфейс, определяющий методы canCast(AbilityContext), onCast(AbilityContext), getCooldownTicks() и getEnergyCost().  
* AbilityContext: Record-класс, содержащий немодифицируемые данные о состоянии: ServerLevel, ServerPlayerEntity, BlockPos (если применимо), и опциональную целевую сущность, определенную сервером.  
* AbilityRegistry: Простой статический реестр, хранящий маппинг строковых идентификаторов на реализации IAbility.

### **Жизненный цикл применения (Cast Lifecycle) и Event Hooks**

Процесс применения способности на сервере проходит через строгий конвейер:

1. **Validation (Валидация):** Проверка аттачмента проклятой энергии. Если энергии недостаточно, каст прерывается.  
2. **Target Selection (Выбор цели):** Сервер выполняет Raycast или Area Query для определения целей.  
3. **Application (Применение):** Вызов метода onCast. Списание энергии, вызов entity.hurtServer()12, наложение меток через setAttached().  
4. **Feedback (Отклик):** Отправка S2C пакета HairpinFxPayload всем игрокам, попадающим в область видимости игрока-кастера (используя PlayerLookup.tracking(player)), для отрисовки кинематографичных эффектов10.

Для пассивных способностей или триггеров используются стандартные Fabric API Event Hooks, такие как AttackEntityCallback или AttackBlockCallback13. Это позволяет внедрять логику (например, восстановление проклятой энергии при базовых ударах) без использования Mixin.

## **5\. Боевая система и Hit Detection (Combat System)**

Определение попаданий (Hit Detection) в 1.21.8 полностью опирается на серверную геометрию. Это исключает возможность использования читов на увеличение дальности атаки (Reach/Hitbox expanders).

### **Райкастинг (Raycast) и Line-of-Sight**

Для мгновенных (hitscan) атак или ближнего боя молотком используется служебный класс ProjectileUtil.raycast(...)15. Использование стандартного entity.raycast() некорректно для боя, так как в последних версиях он обращается только к BlockView.raycast и возвращает BlockHitResult, игнорируя сущности16.  
Алгоритм определения попадания:

1. Сервер вычисляет начальную точку startVec (player.getCameraPosVec(1.0F)).  
2. Вычисляется вектор направления и конечная точка endVec с учетом радиуса действия способности (например, 3.5 блока для молотка, 30 блоков для полета гвоздя).  
3. Формируется Box (ограничивающий объем) от startVec до endVec с расширением expand(1.0D) для учета толщины снаряда и компенсации микро-рассинхронизаций15.  
4. Метод возвращает EntityHitResult ближайшей сущности, удовлетворяющей предикату фильтрации.

### **Фильтрация сущностей и Area Query**

Для способностей массового поражения, таких как Резонанс, используется Area Query. Сервер вызывает level.getEntitiesByClass(LivingEntity.class, box, predicate) для сбора всех сущностей в заданном объеме. Предикат (predicate) фильтрует цели: исключает самого кастера, наблюдателей (isSpectator()), мертвые сущности и союзников (проверяя скорборды или команды).

### **Динамический урон, броня и взаимодействие со средой**

Типы урона полностью дата-ориентированы и загружаются из JSON файлов в директории damage\_type12. Урон типа resonance\_damage не требует сложной регистрации в коде. При попадании сервер создает DamageSource, используя RegistryKey\<DamageType\>, и вызывает entity.hurtServer(serverLevel, damageSource, amount)12.  
Взаимодействие с броней и сопротивлениями управляется через систему тегов. Добавив jujutsumod:resonance\_damage в тег tags/damage\_type/bypasses\_armor.json12, мод гарантирует, что урон по душе игнорирует физическую броню. Отбрасывание (Knockback/Stagger) реализуется программным вызовом target.takeKnockback(...) сразу после успешного применения урона. Если луч райкаста сталкивается с блоком раньше, чем с сущностью, способность может инициировать физическое взаимодействие (разрушение хрупких блоков) через world.breakBlock() или просто визуализировать искры от попадания в стену.

## **6\. Имплементация Нобары Кугисаки (Nobara Implementation Plan)**

Реализация первого геймплейного среза для Нобары требует внедрения специфических механик молотка, гвоздей и техники "Резонанс" (Заколка).

### **Использование Гвоздей (Nails) и Молотка (Hammer)**

* **Гвозди:** В рамках серверного контроля оптимально создать кастомный ProjectileEntity. Гвоздь летит по физической траектории или как hitscan-снаряд. При столкновении с блоком гвоздь остается в нем как декоративная сущность. При столкновении с врагом, гвоздь наносит минимальный урон и устанавливает аттачмент ModAttachments.NAIL\_MARK на цель. Аттачмент сохраняет UUID Нобары и таймстэмп наложения.  
* **Молоток:** При нажатии ЛКМ молоток работает как стандартное оружие ближнего боя. При нажатии ПКМ и наличии гвоздей (проверка Data Component jujutsumod:nail\_count на предмете), клиент отправляет AbilityRequestPayload. Сервер проверяет ресурсы, инициирует анимацию и спавнит Projectile гвоздя.

### **Техника Резонанса и Активация Заколки (Hairpin)**

Заколка — это ультимативная способность, подрывающая все установленные метки.

1. **Триггер:** Игрок активирует способность. Сервер получает запрос и валидирует затраты проклятой энергии.  
2. **Поиск целей (Target Link):** Сервер сканирует мир (в радиусе 100 блоков) на наличие сущностей, у которых активен аттачмент ModAttachments.NAIL\_MARK, проверяя совпадение UUID кастера с UUID владельца метки.  
3. **Авторитарный результат (Server-authoritative result):** Сервер наносит огромный кастомный урон (resonance) всем найденным целям и удаляет метки вызовом removeAttached.  
4. **Визуализация (Client Cinematic Trigger):** Сервер формирует пакет HairpinFxPayload со списком ID пораженных сущностей. Пакет рассылается всем клиентам в зоне видимости.  
5. **Клиентский рендеринг:** Получив пакет, клиент локально спавнит частицы масштабного взрыва проклятой энергии на координатах каждой цели, проигрывает кастомный звук и инициирует эффект дрожания камеры (Camera Shake).

## **7\. Персистентность и управление состоянием (Persistence)**

Эффективное управление памятью критически важно. Оставленные ссылки на сущности приводят к тяжелым утечкам памяти (Memory Leaks), которые способны обрушить сервер (Out of Memory).

### **Временное боевое состояние против Персистентных данных**

Состояния делятся на две категории:

* **Временное состояние:** Аттачменты меток гвоздей, статусы оглушения и текущие кулдауны. Они регистрируются в AttachmentRegistry **без** сериализации (без вызова persistent()) и с флагом copyOnDeath(false). При выгрузке чанка, смене измерения или смерти сущности сборщик мусора (GC) автоматически очистит эти данные8.  
* **Персистентное состояние:** Максимальный запас проклятой энергии, открытые навыки и глобальные анлоки. Эти данные привязываются к AttachmentTarget типа PlayerEntity с указанием persistent(Codec). Это гарантирует их сохранение в NBT-файл игрока на диске. При возрождении игрока эти данные должны быть явно скопированы через обработчик события ServerPlayerEvents.COPY\_FROM8.

Избегание утечек памяти достигается строгим запретом на использование статических коллекций (например, HashMap\<UUID, Entity\>) для хранения ссылок на игроков или мобов. Вся привязка данных осуществляется исключительно через Fabric Attachment API, жизненный цикл которого контролируется самим движком игры.

## **8\. Тестирование и диагностика**

Сложная клиент-серверная архитектура требует надежных инструментов диагностики.

### **Unit-тестирование и Sanity Checks**

Вычислительная логика (расчет кулдаунов, геометрия векторов для Raycast, парсинг Data Components) тестируется через стандартные JUnit/JavaExec тесты без запуска полного клиента Minecraft.  
На этапе инициализации сервера (событие ServerLifecycleEvents.SERVER\_STARTED) запускается блок Sanity Checks. Мод программно проверяет, что все ключи DamageType, DataComponentType и AttachmentType успешно загружены и присутствуют в динамических реестрах. В случае отсутствия критического компонента сервер останавливает загрузку с четким сообщением об ошибке в логе.

### **Инструменты отладки и верификации**

Для тестирования in-game разрабатывается набор дебаг\-команд:

* /jujutsu debug energy set @s 1000 — управление ресурсами.  
* /jujutsu debug mark @e\[distance=..50\] — принудительное наложение меток на сущностей в радиусе для тестирования Заколки.  
* /jujutsu dump state @s — вывод в консоль текущего состояния всех аттачментов сущности.

**Чек-лист внутриигровой верификации (In-game Checklist):**

1. Отправить пакет с подмененными координатами через кастомный клиент (проверка авторитарности сервера — сервер должен проигнорировать координаты).  
2. Заспамить C2S пакеты (проверка Rate Limiter).  
3. Перейти в другое измерение с активными метками (проверка корректной очистки памяти).  
4. Убить моба с меткой (проверка copyOnDeath(false)).

## **9\. Управление рисками (Risk Analysis)**

Процесс разработки сопряжен с рисками, характерными для эволюционирующей экосистемы Fabric.

### **Нестабильность Fabric API и изменения маппингов**

Версии от 1.21.2 до 1.21.8 характеризуются активными изменениями в конвейерах рендеринга и моделях данных18. Изменение обфусцированных маппингов (Yarn) может привести к поломке мода при минорных обновлениях. Стратегия митигации заключается в использовании исключительно публичных интерфейсов Fabric API и полного отказа от Java Reflection.

### **Минимизация Mixin и безопасность мультиплеера**

Использование Mixin для изменения поведения ванильных классов (особенно PlayerEntity и LivingEntity) является частой причиной конфликтов между модами. Архитектура jujutsumod строится на избегании Mixin:

* Добавление данных к сущностям — через Attachment API.  
* Перехват урона — через ServerEntityCombatEvents и динамические теги.  
* Перехват ввода — через регистрацию кастомных пакетов.

Риски производительности от сканирования сущностей (Entity Scans) в мультиплеере высоки. Сканирование радиуса 100 блоков каждый тик уничтожит TPS сервера. Поэтому способность "Резонанс" вызывает level.getEntities() только один раз — строго в момент применения способности. Пограничные случаи мультиплеера, такие как высокий пинг, сглаживаются за счет расширения хитбокса луча в ProjectileUtil и приоритезации серверного времени, при этом клиентская визуализация срабатывает мгновенно (Client Prediction), скрывая сетевые задержки от игрока.

## **10\. Implementation Bible (Архитектурные артефакты)**

Раздел содержит нормативные схемы для реализации первого вертикального среза Нобары.

### **Рекомендованная структура пакетов (Package Structure)**

Структура проекта разделяет клиентскую и серверную логику для предотвращения крашей на выделенном сервере (NoClassDefFoundError).

| Пакет / Директория | Назначение | Ключевые классы |
| :---- | :---- | :---- |
| net.jujutsumod.common | Общая логика (Items, Blocks, Attachments) | JujutsuMod, ModItems, ModAttachments, ModDamageTypes |
| net.jujutsumod.network | Сетевой слой и регистрация Payload | ModNetworkManager, AbilityRequestPayload, HairpinFxPayload |
| net.jujutsumod.ability | Фреймворк способностей | IAbility, AbilityContext, NobaraResonanceAbility |
| net.jujutsumod.combat | Геометрия и расчеты урона | RaycastHitboxUtil, TargetingManager |
| net.jujutsumod.client | Клиент-онли рендеринг, Keybinds (Аннотация @Environment) | JujutsuModClient, AbilityKeybinds, HairpinCinematicRenderer |

### **Диаграмма состояний: Жизненный цикл Резонанса (Hairpin State Machine)**

| Фаза | Исполнитель (Side) | Описание действия | Условие перехода |
| :---- | :---- | :---- | :---- |
| **Idle** | Клиент / Сервер | Ожидание пользовательского ввода. | Игрок нажимает Keybind \-\> **Request**. |
| **Request** | Клиент | Отправка AbilityRequestPayload. Локальный запуск фейкового кулдауна (Prediction). | Пакет уходит на сервер \-\> **Validation**. |
| **Validation** | Сервер | Проверка наличия проклятой энергии, кулдауна и оглушений. | Успех \-\> **Execute**. Провал \-\> **Abort**. |
| **Execute** | Сервер | Area Query по аттачментам NAIL\_MARK. Нанесение DamageSource(resonance). Сброс меток. | Расчет урона завершен \-\> **Sync**. |
| **Sync** | Сервер | Формирование и рассылка HairpinFxPayload и AbilityStateSyncPayload. | Пакеты отправлены \-\> **Cinematic**. |
| **Cinematic** | Клиент | Отрисовка взрыва на координатах целей. Воспроизведение звука. Camera Shake. | Партиклы исчезают \-\> **Idle**. |
| **Abort** | Сервер | Отправка пакета коррекции для отмены ложного предсказания клиента. | Возврат состояния \-\> **Idle**. |

### **Порядок реализации (Vertical Slice Implementation Roadmap)**

Для достижения первого играбельного состояния без переусложнения архитектуры, рекомендуется следовать строгому порядку сборки:  
**Что строить СЕЙЧАС (Первый срез Нобары):**

1. **Инфраструктура:** Зарегистрировать предметы (Молоток, Гвозди), определить DataComponentType для счетчика гвоздей3.  
2. **Состояние:** Внедрить AttachmentRegistry для проклятой энергии и меток гвоздей.  
3. **Сетевой слой:** Настроить передачу C2S AbilityRequestPayload и S2C AbilityStateSyncPayload10.  
4. **Бой (Молоток):** Реализовать базовый ProjectileUtil.raycast для ударов в ближнем бою с серверной авторитарностью15.  
5. **Бой (Гвозди):** Реализовать выстрел NailProjectile, который при попадании вешает ModAttachments.NAIL\_MARK.  
6. **Ультимейт (Резонанс):** Сделать сканирование мира на наличие меток, нанесение кастомного урона из JSON и рассылку HairpinFxPayload для визуализации.

**Что отложить на ПОТОМ (После первого плейтеста):**

1. Комплексные визуальные шейдеры и сложные GUI-деревья способностей.  
2. Продвинутую систему модификаторов урона от брони, зачарований и статусных эффектов (Weakness/Strength).  
3. Разделение хитбоксов на части тела (Headshots/Limb damage).  
4. Сложные предсказания траекторий гвоздей на стороне клиента (Client-side Projectile Prediction).

Данная архитектура гарантирует монолитную безопасность сетевого кода, высокую производительность за счет отказа от глобальных сканирований и модульность системы, что позволит в будущем легко интегрировать новых персонажей и механики на проверенном фундаменте.
